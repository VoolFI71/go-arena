Go Arena — это аллокатор памяти для Go, разработанный для систем, требующих экстремально низких задержек (C10M, HFT, GameDev).

Идея проста: вместо миллионов крошечных аллокаций в куче (heap) и долгих пауз сборщика мусора (GC), мы используем арену со сдвигом указателя (bump-pointer), которая сбрасывается за время O(1). Сборщик мусора перестает замечать "мелкий мусор", и задержка (latency) становится стабильной.

## Бенчмарки

Сравнения проводились на процессоре Intel Core i5-12450H (Windows/amd64). Чем меньше значение, тем лучше.

* Конкурентность: 100 воркеров (workers).

## Примеры использования
### 1. HTTP / TCP сервер (Паттерн Arena Pool)
Это основной сценарий. Используйте `ArenaPool` для переиспользования памяти между запросами.

```go
package main

import (
    "fmt"
    "github.com/VoolFI71/go-arena"
)

// Глобальный пул. 64KB обычно достаточно для большинства HTTP-запросов.
var pool = arena.NewArenaPool(64*1024, 0)

func HandleLog(ctx *fasthttp.RequestCtx) {
    // 1. Получаем арену из пула
    mem := pool.Get()
    defer pool.Put(mem)

    // 2. Выделяем память под структуру внутри арены
    u := arena.New[UserData](mem)

    // 3. Строки создаются без аллокаций в куче!
    u.Name = mem.AllocBytesToString(ctx.PostBody())

    ctx.SetStatusCode(200)
}
```

### 2. Слайсы (MakeSlice + Append)
Демонстрация работы с аналогом `append`, требующим передачи указателя на арену.

```go
// 4. Работа с динамическими данными
func DynamicWork(mem *arena.Arena) {
    // Создаем пустой слайс в арене
    items := arena.MakeSlice[int](mem, 0, 10)

    // Используем arena.Append вместо встроенного append
    for i := 0; i < 100; i++ {
        items = arena.Append(mem, items, i)
    }
    // Даже если размер слайса увеличится в 10 раз, перевыделение памяти произойдет внутри арены
}
```

### 3. Очистка в цикле (Reset)
Полезно для долгих циклов в одной горутине без использования пула.

```go
func Worker() {
    mem := arena.NewArena(1024*1024, 0) // Арена на 1МБ

    for i := 0; i < 1000; i++ {
        processIteration(mem)
        mem.Reset() // Очищаем всё с этой итерации за O(1)
    }
}
```

## Правила безопасности (The Safety Contract)
Ручное управление памятью требует дисциплины. Если кратко:
- **Ограничение области видимости**: не возвращайте указатели на объекты в арене за пределы их жизненного цикла (после вызова `pool.Put` или `Reset`).
- **Конкурентность**: Арена не потокобезопасна. Используйте `ArenaPool` для параллельной работы.
- **Слепота GC (Утечка ссылок)**: Память арены скрыта от сборщика мусора Go. **Никогда** не храните внутри структур арены указатели на объекты из обычной кучи (heap). Иначе сборщик мусора не увидит эти ссылки и удалит оригинальный объект раньше времени, что приведет к падению приложения.
- **Грязная память (Безопасность)**: `Reset()` работает за O(1) и НЕ зануляет память. Вызов `MakeSlice[byte]` может вернуть кусок памяти с данными от прошлого HTTP-запроса (пароли, личные данные). Обязательно перезаписывайте или очищайте эти байты перед отправкой клиенту, чтобы избежать утечек конфиденциальных данных.
- **Независимость данных**: если вам нужны долгоживущие данные, сделайте физическую копию объектов арены (например, сериализацию).

### Антипаттерны (так делать НЕЛЬЗЯ!)
```go
// ОШИБКА: Возврат указателя на память арены
func BadFunction() *User {
    mem := pool.Get()
    defer pool.Put(mem) // Память очищается здесь!

    u := arena.New[User](mem)
    return u // <--- ОПАСНОСТЬ! Вызывающий код получает висячий указатель.
}

// ОШИБКА: Использование одной арены из нескольких горутин
func AsyncLogic(mem *arena.Arena) {
    go func() {
        // Арена не потокобезопасна (кроме Pool).
        // Это приведет к состоянию гонки (data race).
        _ = arena.New[int](mem)
    }()
}

// ОШИБКА: Слепота GC (Хранение указателей кучи внутри арены)
type Node struct {
    Data *string
}
func GCLeak(mem *arena.Arena) {
    node := arena.New[Node](mem)
    str := "Heap Allocated String"
    node.Data = &str // GC не увидит эту ссылку! Переменная 'str' будет удалена.
}
```

### Правильный подход
```go
// Если вам нужно вернуть данные, выполните их копирование или сериализацию
func GoodFunction() []byte {
    mem := pool.Get()
    defer pool.Put(mem)

    u := arena.New[User](mem)
    // ... заполняем u ...

    return json.Marshal(u) // Возвращает независимую от арены копию (slice)
}
```

## Справочник API (API Reference)
### Функции аллокации (Allocators)
- `New[T](a *Arena) *T` — выделяет объект типа T в арене.
- `MakeSlice[T](a *Arena, len, cap int) []T` — создает слайс.
- `AllocString(s string) string` — копирует строку или байты в арену.
- `AllocBytesToString(b []byte) string` — копирует `[]byte` в арену и возвращает строку (`string`).

### Вспомогательные функции (Helper functions)
- `Append(a *Arena, slice []T, items ...T) []T` — эквивалент стандартного `append`, но выделяющий память в арене.

### Управление памятью (Memory management)
- `NewArenaPool(chunkSize, maxRetained int) *ArenaPool` — потокобезопасный пул (рекомендуется для серверов).
- `Reset()` — мгновенная очистка арены (возврат курсора в 0).

## Стабильность API и Версионирование (SemVer)
- Текущий уровень стабильности: **v0** (до 1.0). Ломающие изменения (Breaking changes) все еще возможны.
- Политика SemVer:
  - Patch: исправления багов и внутренние изменения, не ломающие API.
  - Minor: новые дополнения к API без изменения существующих сигнатур.
  - Major (или мажорный релиз до 1.0): обратно-несовместимые изменения API.
- Сигнатуры публичного API автоматически тестируются на предмет компиляции в файле `api_contract_test.go` для защиты от случайных поломок.

## Под капотом (Архитектура)
Go Arena использует связный список (динамический массив) чанков памяти (`[][]byte`).
- При создании вы получаете один чанк (например, 64KB).
- `New[T]` просто сдвигает курсор аллокации вперед внутри текущего чанка.
- Когда чанк заполняется, создается и используется новый чанк.
- Метод `Reset` не очищает память, а лишь скидывает индексы (курсоры) в начало для моментального переиспользования.

> Совет: подбирайте `chunkSize` в `ArenaPool` так, чтобы он покрывал ~90% типичных запросов. Это минимизирует создание новых чанков.
